\chapter{Developing \glc}

\section{The \glc\ Build System}

\glc\ use a GNU Make-based build system. Dependencies are automatically discovered, and extensive meta-programming and preprocessing of source files is undertaken as part of the build. As a result, the build system is complicated. In this section the build system is described and detailed.

\subsection{The Makefile}

Files generated during the build (with the exception of final executables) are written to the directory specified by the \mono{BUILDPATH} environment variable. This is normally \mono{work/build/} (and is set by the \mono{Makefile}) unless a special build configuration is requested.

\subsection{Automatic Discovery}

Interdependencies between source files, together with requirements for auto-generated code are automatically discovered during the build process by processing of source files. All such automatic discovery is described below.

\subsubsection{Code Directive Parsing}\label{sec:buildDiscoveryDirectives}

The modular nature of \glc\ is enabled through the use of directives embedded within the source code (XML documents embedded as comment lines starting with a special tag) which instruct the build system how to link together the various functions. Parsing of these directives is handled by the \mono{codeDirectivesParse.pl}\index{codeDirectivesParse.pl@\mono{codeDirectivesParse.pl}} script.

For \mono{include} directives (which generate files that get included into the source), the script generates \mono{\$(BUILDPATH)/*.xml} files which describe the directive, and \mono{\$(BUILDPATH)/Makefile\_Directives} which contains rules for building those include files.

The script also outputs a file \mono{\$(BUILDPATH)/directiveLocations.xml} which provides list of files that contain each directive. This is used by other scripts to permit rapid processing of files associated with each directive.

\subsubsection{Allocatable Arrays}\label{sec:buildDiscoverAllocatables}

Types and ranks of allocatable arrays required by the \glc\ are discovered by the \mono{allocatableArrays.pl}\index{allocatableArrays.pl@\mono{allocatableArrays.pl}} script. Source files are parsed for dimensionful, allocatable intrinsic types. All such combinations of type and rank are stored to a file \mono{\$(BUILDPATH)/allocatableArrays.xml}. The content of this file is later used to build appropriate memory allocation and deallocation functions (see \S\ref{sec:buildGenerationMemory}).

\subsubsection{Executable Files}\index{files!executable}\index{executable files}\label{sec:buildExecutables}

Files which produce executables (including the \mono{Galacticus.exe} executable) are discovered by the \mono{scripts/build/findExecutables.pl}\index{findExecutables.pl@\mono{findExecutables.pl}} script, which generates rules describing how these files should be built, along with all of their dependencies, and writes them to \mono{\$(BUILDPATH)/Makefile\_All\_Execs}\index{Makefile_All_Execs@\mono{Makefile\_All\_Execs}}. All Fortran files in the \mono{source} directory are parsed, and executable files are identified as those containing a \mono{program} statement. All executables so identified are also added to the list of objects that will be built by \mono{make all}.

\subsubsection{Modules Provided}\index{modules!provided}\index{provided modules}\label{sec:buildModulesProvided}

Determination of which files provide which Fortran modules is carried out by the \mono{scripts/build/moduleDependencies.pl}\index{moduleDependencies.pl@\mono{moduleDependencies.pl}} script, which generates rules describing these dependencies (and how to build the module file by compiling the corresponding source file) and writes them to \mono{\$(BUILDPATH)/Makefile\_Module\_Dependencies}\index{Makefile_Module_Dependencies@\mono{Makefile\_Module\_Dependencies}}.

Additionally, rules are generated which describe how to build the following classes of file:
\begin{description}
\item[\mono{*.mod.d}] Contains a list of all object files upon which the module depends. Used in constructing the final set of objects which must be linked to build an executable.
\item[\mono{*.mod.gv}] Contains a list of all source files upon which the module depends. Used in constructing \gls{graphviz} visualizations of dependencies.
\item[\mono{*.m}] Contains a list of all modules provided by the corresponding object file. Used when building object files to test whether corresponding module files have been changed---allows avoidance of recompilation cascades if modules do not need to be updated (i.e. if the modules do not differ as judged by the \mono{scripts/build/compareModules.pl}\index{findExecutables.pl@\mono{compareModules.pl}} script).
\end{description}

\subsubsection{Modules Used}\index{modules!used}\index{used modules}\label{sec:buildModulesUsed}

Determination of which files use which Fortran modules is carried out by the \mono{scripts/build/useDependencies.pl}\index{useDependencies.pl@\mono{useDependencies.pl}} script, which generates rules describing these dependencies and writes them to \mono{\$(BUILDPATH)/Makefile\_Use\_Dependencies}\index{Makefile_Use_Dependencies@\mono{Makefile\_Use\_Dependencies}}.

Additionally, rules are generated which describe how to build the following classes of file:
\begin{description}
\item[\mono{*.d}] Contains a list of all object files upon which each object file depends. Used in constructing the final set of objects which must be linked to build an executable.
\item[\mono{*.gv}] Contains a list of all source files upon which each file depends by virtue of \mono{use} statements. Used in constructing \gls{graphviz} visualizations of dependencies.
\item[\mono{*.fl}] Contains a list of libraries upon which each object file depends. Used in constructing the final set of libraries which must be linked with the executable.
\end{description}

\subsubsection{Included Files}\index{files!include}\index{include files}\label{sec:buildIncludeDeps}

Dependencies on files due to the use of ``\mono{include}'' statements (in both Fortran and C source files) are discovered by the \mono{scripts/build/includeDependencies.pl}\index{includeDependencies.pl@\mono{includeDependencies.pl}} script, which generates rules describing these dependencies and writes them to \mono{\$(BUILDPATH)/Makefile\_Include\_Dependencies}\index{Makefile_Include_Dependencies@\mono{Makefile\_Include\_Dependencies}}.

All Fortran and C/C++ source files in the \mono{source} directory are parsed. Files specified in any include statement in a source file are added as a dependency of that source file if unless the source file is C/C++ and the named include file can be found in either \mono{source/}, a standard system include path, or in any include path specified in the \mono{GALACTICUS\_CFLAGS}\index{environment variables!GALACTICUS_CFLAGS@\mono{GALACTICUS\_CFLAGS}}\index{GALACTICUS_CFLAGS@\mono{GALACTICUS\_CFLAGS}} or \mono{GALACTICUS\_CPPFLAGS}\index{environment variables!GALACTICUS_CPPFLAGS@\mono{GALACTICUS\_CPPFLAGS}}\index{GALACTICUS_CPPFLAGS@\mono{GALACTICUS\_CPPFLAGS}} environment variables.

\subsubsection{Parameter Dependencies}\index{parameters!dependencies}\label{sec:parameterDependencies}

All runtime parameters upon which a given executable may depend are discovered by the \mono{scripts/build/parameterDependencies.pl}\index{parameterDependencies.pl@\mono{parameterDependencies.pl}} script, which generates XML files listing these parameters and writes them to \mono{\$(BUILDPATH)/\textless executableName\textgreater.parameters.xml}\index{*.parameters.xml@\mono{*.parameters.xml}}. All Fortran and C++ files in the \mono{source} directory are parsed, and embedded parameter definitions (i.e. embedded XML blocks in lines beginning ``!\@'' for Fortran or ``//\@'' for C++ with root element \mono{inputParameter}) are identified. Any files included into a source file are also searched. In the case of Fortran source files, \mono{source/*.F90}, the corresponding preprocessed file, \mono{\$(BUILDPATH)/*.p.F90}, will be searched for parameter dependencies.

\subsection{Code Generation}

During build, code is generated automatically. Much of this is to link together functions within \glc\ (thereby permitting the modular nature of \glc), but also includes generation of various utility functions. All code generation is described below.

\subsubsection{Memory Management Functions}\label{sec:buildGenerationMemory}

Functions for allocation and deallocation of intrinsic-type arrays are generated by the \mono{memoryManagementFunctions.pl}\index{memoryManagementFunctions.pl@\mono{memoryManagementFunctions.pl}} script. Utilizing the list of such types in the \mono{\$(BUILDPATH)/allocatableArrays.xml} file (see \S\ref{sec:buildDiscoverAllocatables}) overloaded functions are created for each type. These memory management functions keep track of the total amount of memory allocated.

\mono{memoryManagementFunctions.pl}\index{memoryManagementFunctions.pl@\mono{memoryManagementFunctions.pl}}
 
\subsubsection{Directives}\label{sec:codeBuildIncludeDirectives}

Generation of code to implement the functionality of \mono{include} directives is carried out by the \mono{buildCode.pl}\index{buildCode.pl@\mono{buildCode.pl}} script. This script simply reads an XML file generated by the \mono{codeDirectivesParse.pl} (see \S\ref{sec:buildDiscoveryDirectives}), calls the appropriate function to generate the necessary code, passes this through the preprocessor (see \S\ref{sec:sourceTreePreprocessor}), and outputs the result to the appropriate include file.

\subsubsection{Pre-processing and Post-processing}\label{sec:codeBuildPrePostProcess}

Before being compiled, all Fortran source files are passed through a tree-based preprocessor. The functionality of this preprocessor is described in \S\ref{sec:sourceTreePreprocessor}. The preprocessor is invoked on each source file by the \mono{preprocess.pl}\index{preprocess.pl@\mono{preprocess.pl}} script, which takes a \mono{source/*.F90} file as input and outputs a preprocessed file \mono{\$(BUILDPATH)/*.p.F90}.

Output from the \mono{gfortran} compiler is directed through the \mono{postprocess.pl}\index{postprocess.pl@\mono{postprocess.pl}} script. This script currently performs the following functions:
\begin{itemize}
\item Builds a map from line numbers in the preprocessed source file back to line numbers in the unpreprocessed source, and translates line numbers in any error or warnings messages from the compiler into line numbers in the unpreprocessed source;
\item Filters out incorrect warning messages about the lack of scalar finalizers (see GCC \href{https://gcc.gnu.org/bugzilla/show_bug.cgi?id=58175}{PR58175});
\item Checks for unused function attributes in the source and filters out warnings about these unused functions.
\end{itemize}

All files involved in the build process are summarized below.

\begin{description}

\item[\mono{scripts/build/allocatableArrays.pl}\index{allocatableArrays.pl@\mono{allocatableArrays.pl}}:] Discovers dimensionful, allocatable, intrinsic-typed arrays in the source code, and generates a summary of their types and ranks to the file \mono{\$(BUILDPATH)/allocatableArrays.xml} (see \S\ref{sec:buildDiscoverAllocatables});
  
\item[\mono{scripts/build/buildCode.pl}\index{buildCode.pl@\mono{buildCode.pl}}:] Acts upon \mono{include} directives embedded in the source code and generates the corresponding include file (see \S\ref{sec:codeBuildIncludeDirectives});

\item[\mono{scripts/build/codeDirectivesParse.pl}\index{codeDirectivesParse.pl@\mono{codeDirectivesParse.pl}}:] Discovers and parses directives embedded in source files. \item[\mono{scripts/build/compareModuleFiles.pl}\index{compareModuleFiles.pl@\mono{compareModuleFiles.pl}}:] Tests whether two Fortran module files are identical (ignoring timestamp information). Used to avoid updating modules when not necessary and so avoids recompilation cascades;

\item[\mono{scripts/build/includeDependencies.pl}\index{includeDependencies.pl@\mono{includeDependencies.pl}}:] Discovers (and generates rules for) dependencies between files arising from the use of ``\mono{include}'' statements (see \S\ref{sec:buildIncludeDeps});

\item[\mono{\normalfont \ttfamily scripts/build/findExecutables.pl\index{findExecutables.pl@{\normalfont \ttfamily findExecutables.pl}}}:] Discovers (and generates rules for) files which generate executables (see \S\ref{sec:buildExecutables});

\item[\mono{scripts/build/libraryDependencies.pl}\index{libraryDependencies.pl@\mono{libraryDependencies.pl}}:] Determines the set of libraries that must be linked with each executable, and ensures they are ordered correctly for static linking;

\item[\mono{scripts/build/moduleDependencies.pl}\index{moduleDependencies.pl@\mono{moduleDependencies.pl}}:] Discovers (and generates rules for) dependencies of module files on their source file statements (see \S\ref{sec:buildModulesProvided});

\item[\mono{\$(BUILDPATH)/Makefile\_All\_Execs}\index{Makefile_All_Execs@\mono{Makefile\_All\_Execs}}:] Contains rules describing how to build the executable file for each distinct program---generated by \mono{scripts/build/findExecutables.pl}\index{findExecutables.pl@\mono{findExecutables.pl}} (see \S\ref{sec:buildExecutables});

\subsection{Profiling the Build}\index{profiling!build}\index{build profiling}

Tooling is provided to help profile the build process. This is useful to identify bottlenecks during build. Profiling can be switched on using the \mono{compileprof} build option. For example:
\begin{verbatim}
make -j16 GALACTICUS_BUILD_OPTION=compileprof Galacticus.exe >& build.log
\end{verbatim}
Every command run by \mono{Make} will be timed and the timing results output. In the above these outputs are collected to the \mono{build.log} file. Timing result lines begin \mono{++Task:} followed by the start and end times of the task in, and then the command run. While you can look through these manually a script is provided that generates a report from these timing data as a web page. For example:
\begin{verbatim}
./scripts/build/buildProfiler.pl build.log profile.html --durationMinimum 10
\end{verbatim}
will parse timing data from \mono{build.log} and generate a web page \mono{profile.html} showing the results. In this case the \mono{--durationMinimum 10} option specifies that only commands which took 10 or more seconds to run should be included in the report. (Reducing this limit will lead to a very long report.)

The profile report begins by listing the total time taken for the build. Following that is a list of all commands performed ordered by \emph{completion} time. Next to each command is a bar which extends from the start to the end time of the command. Each second of the bar is colored to indicate the degree of build parallelism at that time---green shows maximum parallelism, while red shows no parallelism (i.e. a single command was running at that time).

Finally, each command is assigned a ``cost'', $\chi$. This is defined as:
\begin{equation}
  \chi = \sum_{i_\mathrm{start}}^{i_\mathrm{end}} N_i^{-1},
\end{equation}
where $i_\mathrm{start}$ and $i_\mathrm{end}$ are the start and end times of the command, and $N_i$ is the number of commands running in parallel at time $i$. Cost is therefore higher for commands which run longer and for commands which are executed with less parallelism. A ranked list of commands, from most to least costly is included in the report.
